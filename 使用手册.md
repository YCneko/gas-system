# Gas System — 面向化工园区的多源废气智能治理系统

## 使用手册

---

## 一、系统概述

Gas System 是一套面向化工园区的多源废气智能治理系统原型，集成了**多源数据融合**、**VOCs浓度预测**、**可视化监控看板**三大核心功能。采用前后端分离架构，支持 Docker 一键部署。

### 核心功能

| 模块 | 功能 | 技术栈 |
|------|------|--------|
| 数据融合 | JSON/CSV/Excel 多格式接入、数据清洗、分钟级时序对齐 | Python + Pandas |
| 智能预测 | VOCs 未来 6 小时趋势预测 (准确率 ≥75%) | TensorFlow LSTM |
| 预警引擎 | 80mg/m³ 阈值检测，高/中/低三级预警，≤10 秒触发 | Flask + MySQL |
| 监控看板 | 实时数据、预测曲线、预警列表，响应式布局 | Vue 3 + ECharts |
| 报告导出 | 历史排放数据 CSV 下载 | Flask Response |

## 二、环境要求

### Docker 部署 (推荐)

- Docker Desktop 或 Docker Engine 20.10+

## 三、Docker 一键部署

### 3.1 启动服务

```bash
cd Gas System
docker-compose up -d --build
```

此命令会执行：
1. 拉取基础镜像 (Python 3.13-slim, Node 18-alpine, Nginx alpine, MySQL 8.0)
2. 构建前端 (Vue编译 → Nginx静态文件)
3. 构建后端 (安装依赖含 TensorFlow → 启动 Flask)
4. 启动 MySQL (自动建库建表)
5. 健康检查 → 按依赖顺序启动

**首次构建约 5-10 分钟**（TensorFlow 约 350MB），后续启动仅需数秒。

### 3.2 验证部署

```bash
# 检查容器状态
docker ps

# 应看到 3 个容器:
#   gas_frontend    Up   0.0.0.0:80->80/tcp
#   gas_backend     Up   0.0.0.0:5000->5000/tcp
#   gas_mysql       Up   0.0.0.0:3307->3306/tcp (healthy)
```

### 3.3 访问系统

| 服务 | 地址 |
|------|------|
| 前端监控看板 | http://localhost |
| 后端 API | http://localhost:5000 |
| API (通过前端代理) | http://localhost/api/* |

### 3.4 停止与清理

```bash
# 停止所有服务
docker-compose down

# 停止并删除数据卷 (重置数据库)
docker-compose down -v
```

## 四、本地开发部署

### 4.1 数据库准备

确保 MySQL 8.0 运行在 `localhost:3306`，创建数据库：

```sql
CREATE DATABASE IF NOT EXISTS gas_system
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_unicode_ci;
```

### 4.2 后端启动

```bash
cd backend

# 创建虚拟环境 (首次)
python -m venv .venv
.venv\Scripts\activate     # Windows
# source .venv/bin/activate  # Linux/Mac

# 安装依赖
pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple

# 生成真实感数据并导入数据库
python generate_real_data.py

# 运行数据分析 (生成 scaler + 热力图)
python data_analysis.py

# 训练 LSTM 模型 (生成模型文件)
python train_model.py

# 启动 Flask
python app.py
```

### 4.3 前端启动

```bash
cd frontend

# 安装依赖
npm install --registry=https://registry.npmmirror.com

# 启动开发服务器
npm run serve
```

开发服务器运行在 `http://localhost:8080`，会自动代理 `/api` 到后端 `http://127.0.0.1:5000`。

## 六、数据管理

### 6.1 支持的格式

| 格式 | 说明 |
|------|------|
| CSV | 逗号分隔，UTF-8/GBK 编码 |
| Excel | .xlsx / .xls, openpyxl 解析 |
| JSON | 实时流式数据，POST body |

### 6.2 数据表结构

**emission_data** — 废气排放浓度

| 字段 | 类型 | 说明 |
|------|------|------|
| sensor_id | VARCHAR(50) | 传感器编号 |
| timestamp | DATETIME | 采样时间戳 |
| voc_concentration | FLOAT | VOCs浓度 (mg/m³) |
| nox_concentration | FLOAT | NOx浓度 (mg/m³) |
| so2_concentration | FLOAT | SO₂浓度 (mg/m³) |

**weather_data** — 气象数据

| 字段 | 类型 | 说明 |
|------|------|------|
| station_id | VARCHAR(50) | 气象站编号 |
| timestamp | DATETIME | 观测时间戳 |
| temperature | FLOAT | 温度 (°C) |
| humidity | FLOAT | 相对湿度 (%) |
| wind_speed | FLOAT | 风速 (m/s) |
| wind_direction | FLOAT | 风向 (度) |

**equipment_data** — 设备运行参数

| 字段 | 类型 | 说明 |
|------|------|------|
| equipment_id | VARCHAR(50) | 设备编号 |
| timestamp | DATETIME | 记录时间戳 |
| operating_load | FLOAT | 运行负荷 (%) |
| production_phase | VARCHAR(50) | 生产时段 |
| status | VARCHAR(20) | 设备状态 |

**alert_records** — 预警记录

| 字段 | 类型 | 说明 |
|------|------|------|
| alert_timestamp | DATETIME | 预警生成时间 |
| predicted_exceedance_time | DATETIME | 预测超标时间 |
| predicted_value | FLOAT | 预测超标值 (mg/m³) |
| alert_level | VARCHAR(10) | 预警等级: 高/中/低 |

### 6.3 数据生成器

当没有真实数据时，可使用内置生成器：

```bash
cd backend
python generate_real_data.py
```

生成 90 天 × 24 小时的真实感数据，基于物理规律模拟 VOCs 与温度/湿度/风速/负荷的相关性。

## 七、模型训练

### 7.1 训练流程

```bash
cd backend

# 步骤 1: 数据分析 + 特征工程
python data_analysis.py
# 输出: correlation_heatmap.png, scaler.pkl, engineered_features.npz

# 步骤 2: 模型训练
python train_model.py
# 输出: vocs_lstm_model.keras, best_model.keras, training_history.png
#       模型结构图, 预测对比图, 评估指标
```

### 7.2 模型参数

| 参数 | 值 |
|------|------|
| 架构 | LSTM(128) → Dropout(0.3) → LSTM(64) → Dropout(0.3) → Dense(32) → Dense(6) |
| 参数量 | 124,902 |
| 输入 | 24 小时 × 14 维特征 |
| 输出 | 未来 6 小时 VOCs 浓度 |
| 损失函数 | MSE |
| 优化器 | Adam (lr=0.001) |
| 正则化 | Dropout(0.3) + EarlyStopping(patience=15) |
| 数据划分 | 训练 70% / 验证 15% / 测试 15% (时序划分) |

### 7.3 性能指标

| 指标 | 要求 | 实测 |
|------|------|------|
| 准确率 (1-MAPE) | ≥75% | 82.61% |
| MAE | — | 12.40 mg/m³ |
| 单次推理 | ≤2s | <100ms |
| 预警触发 | ≤10s | <1s |

## 九、常见问题

### Q1: Docker 构建失败，提示 "502 Bad Gateway"

**A**: 国内网络访问 Docker Hub 和 Debian 仓库可能不稳定。已配置清华镜像加速 pip，如仍有问题可配置 Docker 镜像加速器。

### Q2: 预测接口返回 "有效数据不足"

**A**: 数据库需要至少 30 条小时级历史记录。运行 `generate_real_data.py` 生成数据后重新导入。

### Q3: 预测值全部为 0 或相近

**A**: Docker MySQL 是新实例，数据为空。需先导入数据：
```bash
docker exec -i gas_mysql mysql -uroot -p123456 gas_system < data.sql
```

### Q4: 模型文件缺失

**A**: 运行 `python train_model.py` 训练模型。Docker 部署时如模型不存在，容器启动会自动训练（约需 2-3 分钟）。

### Q5: "cryptography package is required" 错误

**A**: MySQL 8.0 使用 caching_sha2_password 认证，需要 cryptography 包。已在 requirements.txt 中添加，确保运行 `pip install -r requirements.txt`。

### Q6: 端口冲突

**A**: 默认端口 5000, 80, 3307。如有冲突，修改 `docker-compose.yml` 中的端口映射。

### Q7: 如何更换为真实数据

**A**: 将真实数据整理为以下格式后上传：
- 排放数据: sensor_id, timestamp, voc_concentration, nox_concentration, so2_concentration
- 气象数据: station_id, timestamp, temperature, humidity, wind_speed, wind_direction
- 设备数据: equipment_id, timestamp, operating_load, production_phase, status

支持 CSV/Excel 格式，通过 `/api/data/upload` 接口导入。

### Q8: 如何提升模型准确率

**A**: 当前准确率 82.61%。进一步提升建议：
1. 增加训练数据量 (当前 2160 条, 建议 ≥5000 条)
2. 延长回看窗口 (当前 24h, 可尝试 48h/72h)
3. 增加 LSTM 单元数 (当前 128→64, 可升级到 256→128)
4. 添加更多气象特征 (气压、降雨量等)
5. 调整学习率策略 (余弦衰减)
